

# Исправление уязвимости: доступ до одобрения

## Проблема
Пользователь после регистрации через `register-request` сразу получает валидную сессию (email_confirm: true) и может обойти экран ожидания, зайдя напрямую на любой маршрут. `AppLayout` не проверяет статус `approved`.

## Изменения

### 1. `AppLayout.tsx` — блокировка неодобренных
Добавить проверку `profile?.approved === false` → редирект на `/auth` (где показывается экран ожидания). Аналогично если `profile` ещё null но user есть — показывать загрузку, не контент.

```
if (!user) return <Navigate to="/auth" replace />;
if (!profile) return <загрузка>;
if (!profile.approved) return <Navigate to="/auth" replace />;
```

### 2. RLS-политики (опционально, для глубокой защиты)
Сейчас все SELECT-политики проверяют только `authenticated`. Можно добавить проверку `approved` через функцию `is_approved(auth.uid())`, но это затронет ~15 таблиц и усложнит систему. Рекомендую начать с фикса на уровне UI (п.1), так как данные всё равно доступны только через клиент.

### Затронутые файлы
- `src/components/AppLayout.tsx` — добавить проверку approved

### Объём
Минимальный — 3-5 строк кода в одном файле.
