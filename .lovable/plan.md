
# Исправление перегенерации секций: убирать неподтверждённые факты, а не только просить об этом

## Do I know what the issue is?
Да.

## Что именно сломано сейчас
Проблема двойная:

1. `generate-partner-profile/index.ts` при перегенерации секции **только подмешивает результаты фактчекинга в промт**, но:
   - **не запускает фактчекинг повторно** для новой версии секции;
   - **не обновляет** `generated_from_sources_json.verification` и `verification_summary` в профайле.

2. `PartnerProfileTab.tsx` после перегенерации продолжает показывать **старые verification-данные**:
   - UI берёт `verificationData`, если оно уже есть в state;
   - после `regenerateSection()` этот state **не заменяется свежими данными** по секции.

Из-за этого даже после команды “не выводить плохие результаты фактчекинга”:
- в тексте могут остаться слабые факты;
- жёлтые предупреждения в интерфейсе часто остаются прежними, потому что это **старый результат проверки**, а не новая оценка.

## Что нужно сделать

### 1. Backend: сделать перегенерацию секции с обязательной самопроверкой
Файл: `supabase/functions/generate-partner-profile/index.ts`

В ветке `if (section_key && profile_id)`:

- после генерации секции запускать **fact-check только для этой секции**;
- если есть `unconfirmed`/`contradicted`, выполнять **repair pass**:
  - передать новой итерации только проблемные факты;
  - жёсткое правило: если факт не подтверждается источниками, **удалить его**, а не перефразировать;
  - если подтверждённых данных не хватает — вернуть `Данные не найдены`.

После repair pass:
- ещё раз прогонять секцию через fact-check;
- сохранять в БД:
  - обновлённый текст секции;
  - `references_json`;
  - обновлённый `generated_from_sources_json.verification` для этой секции;
  - обновлённый `generated_from_sources_json.verification_summary`.

В response возвращать:
- `section_content`
- `section_verification`
- `verification_summary`
- `references`
- при необходимости `remaining_issues`

### 2. Frontend: сразу подменять проверку по секции на новую
Файл: `src/components/partner/PartnerProfileTab.tsx`

В `regenerateSection()`:
- после успешного ответа обновлять `form[sectionKey]`;
- **заменять verification именно для этой секции** в `verificationData`;
- пересчитывать summary из свежих данных;
- если backend вернул, что проблемы остались, показывать не просто success, а предупреждение вида:
  - `Секция обновлена, но осталось 1 неподтверждённое утверждение`.

Также важно убрать зависимость от устаревшего state:
- либо всегда синхронизировать `verificationData` с ответом backend;
- либо после перегенерации сбрасывать локальный verification-state и читать свежий `draftProfile`.

### 3. Усилить промт для секционной перегенерации
Файл: `supabase/functions/generate-partner-profile/index.ts`

В prompt добавить жёсткие правила:
- не сохраняй факты со статусом `unconfirmed` или `contradicted`;
- не замещай отсутствие доказательств “аналитичной формулировкой”;
- не повторяй спорный факт в смягчённой форме;
- если пользователь просит убрать плохие результаты фактчекинга — это приоритетная инструкция.

Прямо сформулировать:
`Если факт нельзя подтвердить по источникам, он не должен попасть в итоговый текст.`

### 4. Сохранить консистентность между текстом и бейджами
Сейчас текст и бейджи живут отдельно. Нужно сделать так, чтобы после regen:
- текст секции,
- детализация проверки,
- summary наверху,
- цветной badge у секции

обновлялись из **одного и того же нового результата проверки**.

## Файлы
| Файл | Изменение |
|------|-----------|
| `supabase/functions/generate-partner-profile/index.ts` | Повторный фактчекинг секции, repair pass, обновление verification в `generated_from_sources_json`, возврат свежей проверки в response |
| `src/components/partner/PartnerProfileTab.tsx` | Обновление `verificationData` после regen, корректный toast/status, синхронизация бейджей и деталей |

## Результат после исправления
После нажатия “Перегенерировать” с комментарием вроде “не выводить неподтверждённые факты” система должна:
1. перегенерировать секцию;
2. сама проверить её заново;
3. удалить неподтверждённые утверждения или пометить, что данных нет;
4. сразу показать обновлённые бейджи и детали проверки без старых предупреждений.

## Как я проверю после реализации
- воспроизведу кейс с секцией `key_events_and_touchpoints`;
- перегенерирую с комментарием на исключение неподтверждённых фактов;
- проверю, что:
  - текст реально изменился;
  - спорный факт исчез или заменён на “Данные не найдены”;
  - `section_verification` обновился;
  - warning badge уменьшился или пропал;
  - summary сверху совпадает с новым результатом.
