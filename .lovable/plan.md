

# Корень проблемы: несоответствие ссылок в PDF

## Причина

Предыдущий фикс добавил `seqNum = ri + 1` — перенумерацию источников с 1. Но **inline-ссылки в тексте** (`[1]`, `[5]`, `[10]`) остались с оригинальными номерами от LLM. Если в `references_json` было 10 источников с номерами 1–10, но один пропущен (например, нет записи с number=4), то:

- Источник с `number=10` (ЛЭТИ) стал `[9]` в списке
- Но текст по-прежнему ссылается на `[10]`

**Это чисто баг PDF-экспорта**, внесённый предыдущим фиксом.

## Решение

**Не перенумеровывать** источники. Использовать оригинальный `ref.number` из данных. Inline-цитаты в тексте профайла уже согласованы с этими номерами — именно так их сгенерировал LLM.

Конкретно в `ProfilePdfExport.tsx`:
- Строка 433: заменить `const seqNum = ri + 1` на `const seqNum = ref.number ?? (ri + 1)` — использовать оригинальный номер, fallback на последовательный только если `number` отсутствует.
- Строка 444: уже использует `seqNum` — ничего менять не нужно.

Одно изменение, одна строка.
